## Collision-layer scheme actually used

Full detail in `components/LAYERS.md`. Summary (named in `project.godot`
`[layer_names]`):

| Bit | Value | Name          | Node                | mask           | monitoring / monitorable |
|-----|-------|---------------|---------------------|----------------|--------------------------|
| 1   | 1     | world         | (reserved)          | -              | -                        |
| 2   | 2     | player_body   | Player body         | 1 (world)      | -                        |
| 3   | 4     | enemy_body    | Enemy body          | 1 (world)      | -                        |
| 4   | 8     | player_hitbox | Sword (Hitbox)      | 0              | false / true             |
| 5   | 16    | enemy_hurtbox | Hurtbox             | 8              | true  / false            |

One-way discipline: only the Hurtbox monitors (mask includes layer 8); the Sword
Hitbox is monitorable but monitors nothing (mask 0). Exactly one `area_entered`
fires per overlap, on the Hurtbox. The Sword's CollisionShape2D starts disabled;
`attack()` rotates the Sword to the facing angle and enables the shape for 0.15 s.

## Recipes that were awkward/incomplete in practice

1. **health-and-damage.md shows only an always-live hitbox (fireball), never a
   time-windowed melee swing.** The recipe's Hitbox is a persistent projectile;
   it never demonstrates *activating* a hitbox for a swing window. The mechanism
   that makes a slash land: start the Hitbox's `CollisionShape2D` with
   `disabled = true`, then set `disabled = false` for the attack window.
   Enabling a shape that already overlaps the monitoring Hurtbox makes the
   Hurtbox emit `area_entered` on the *next physics frame* -- verified here.
   *Fix:* add a "Melee swing" variation to the recipe showing the disabled-shape
   toggle plus rotating the Hitbox to `facing.angle()`.

2. **The recipe wires `@export var health: HealthComponent` "in the Inspector".**
   For code-first hand-authoring, serializing a node-reference export across an
   instanced-scene boundary needs the fiddly `node_paths` field in `.tscn`.
   Cleaner for code-first: the parent assigns it in `_ready()`
   (`_hurtbox.health = _health`), matching the "call down, signal up" rule from
   scene-composition.md. *Fix:* the recipe should show the parent-wires-in-code
   form as the primary path for GUI-less projects.

3. **One-slash-kill needs `damage >= max_health` because i-frames + entry-only
   detection mean a single `area_entered` fires per swing.** `area_entered` only
   triggers on entry, not while the areas keep overlapping, so a 0.15 s window
   lands exactly one hit. Enemy `max_health` is set to 3 and Sword `damage` to 3
   so the milestone's asserted death happens in one slash. Not a recipe bug, but
   worth stating: with `invincibility_time` > 0 a stationary overlap will not
   re-tick. (health-and-damage.md notes this for hazards but not for melee.)

4. **Milestone spec item #3 wording ("the sword's area_entered deals damage")
   vs the recipe's direction (the Hurtbox detects).** Resolved toward the recipe
   -- "follow the Hitbox->Hurtbox one-way layer discipline from the recipe" means
   the Hurtbox is the monitor. Behavior is identical from the player's view; the
   smoke assertions are direction-agnostic (health decreased + died).

Otherwise topdown-movement.md and scene-composition.md were accurate as written
(`Input.get_vector` un-normalized, `move_toward` accel/friction, component-as-
child-scene composition all worked first try).

## The 3-hit combo (design + tunables)

Press attack within `combo_window` of the last swing to chain; miss the window and
the next attack restarts at hit 1; after hit 3 it wraps back to hit 1.

| Export         | Value  | Role                                                   |
|----------------|--------|--------------------------------------------------------|
| arc_degrees    | 120.0  | total sweep of an arc hit (+/- 60 deg of facing)       |
| swing_duration | 0.2 s  | length of one swing (arc tween, or lunge hold)         |
| combo_window   | 0.5 s  | grace window to continue the combo (ComboResetTimer)   |
| lunge_impulse  | 140.0  | forward impulse fed to `_knockback` on hit 3           |

- **Hit 0 (arc A):** `SwordPivot.rotation` set to `facing.angle() + 60deg`, blade
  enabled + shown, then **tweened to `facing.angle() - 60deg`** over `swing_duration`.
- **Hit 1 (arc B):** same but `-60deg -> +60deg` (the opposite sweep).
- **Hit 2 (lunge):** pivot set straight to `facing.angle()`, blade enabled + shown,
  `_knockback = facing * lunge_impulse` (reuses the decaying-impulse system so the
  player **slides forward ~8-9 px and stops**), held for `swing_duration`.
- After each swing: `_combo_index = (_combo_index + 1) % 3`, `_attacking = false`,
  `ComboResetTimer.start(0.5)`. On timeout -> `_combo_index = 0`.
- One sweep = **one hit per enemy** (Hurtbox is entry-only + i-frames) but **cleaves
  multiple enemies**. Intended. Because the arc passes through the exact facing
  direction mid-sweep, a target directly in front is still cleanly hit -- no test
  positions needed to move.

## Enemy death sequence (timings)

Does NOT pause the tree. On `died` (latched by `_dying`):
1. `is_dead = true` (the `_physics_process` early-return halts AI/movement).
2. **Pass-through:** `$CollisionShape2D.set_deferred("disabled", true)` and
   `_hurtbox.set_deferred("monitoring", false)`; also disable the attack-hitbox shape.
3. **Jerk-stop lurch:** tween `global_position` to
   `global_position + _last_hit_dir * death_lurch_distance` (26 px) over **0.12 s**
   with `TRANS_QUINT` + `EASE_OUT` -- shoots out and stops abruptly. `_move_velocity`,
   `_knockback`, `velocity` zeroed first so they do not fight the tween. (Deliberately
   NOT the gradual `_knockback` decay.)
4. **Blink:** looping tween toggling `$Body.visible` every 0.08 s, run for **0.35 s**.
5. `queue_free()`. Distinct from the player's circle-pop death.

## Recipes that were awkward/incomplete in practice

1. **health-and-damage.md's "Time-windowed melee swing" variation covers only a
   FIXED disabled-shape toggle, never a moving/arc hitbox.** It shows
   `_sword_shape.disabled = false; await create_timer(0.15).timeout; disabled = true`.
   The arc combo instead **rotates a pivot with a tween across the window** and awaits
   `tween.finished` rather than a timer. *Fix:* add an "arc sweep on a pivot" note:
   parent the hitbox under a `Node2D` pivot at the attacker origin, offset the blade
   `+x`, and `tween SwordPivot.rotation` from `facing +arc/2` to `facing -arc/2` while
   the shape is enabled. A mid-arc sample crosses the facing direction, so a target
   dead-ahead is still hit.

2. **The awaited-swing guard latches until the tween's `finished`, and a caller that
   stops waiting mid-swing leaves it latched.** With `attack()` awaiting
   `tween.finished`, `_attacking` stays true until the tween actually completes. A
   fire-and-forget caller that breaks its own wait loop early (our smoke leg (e)
   breaks the instant the dummy takes damage, ~6 frames < the 12-frame 0.2 s tween)
   sees `_attacking` still true on the next call, which then no-ops on the guard.
   *Fix:* the recipe should note that with a tween-driven swing the attacking guard
   is released on `tween.finished` (not on the shape-disable line), so callers that
   need a clean state must drain in-flight swings (`while attacker._attacking: await
   physics_frame`) -- exactly what leg (f) now does before starting the combo.

3. **For a hard, abrupt-stop lurch (enemy death), the recipe's decaying-`_knockback`
   variation is the wrong tool.** The decaying impulse eases out over many frames; a
   defeated-enemy "jerk stop" wants to shoot out and halt. We tween `global_position`
   directly with `TRANS_QUINT`/`EASE_OUT` over 0.12 s and zero `_move_velocity` /
   `_knockback` / `velocity` first so they don't fight it. *Fix:* add a short note to
   the Knockback variation distinguishing "sustained shove" (decaying impulse) from
   "impact lurch/jerk-stop" (short position tween, ease-out), and to null the motion
   velocities so the two systems don't compound.

wear is a separate number (power vs hardness, three bands). HP damage and wear never

## The two-system split (one chokepoint, two resolvers)

- **System 1 -- combat HP.** `CombatResolver.hp_damage(atk, def) -> int` = `maxi(1, atk - def)`.
  Subtractive, floored at 1. DEF mitigates; hardness NEVER gates enemy HP (flesh is never
  "too hard").
- **System 2 -- durability wear.** `DurabilityResolver.resolve(power, hardness, threshold,
  wear_max) -> Dictionary` returns `{ "weapon_wear": int, "affects_target": bool }` for
  `over = hardness - power`:
  - Band A (`over <= 0`): weapon_wear 0, affects_target true
  - Band B (`0 < over <= threshold`): weapon_wear `ceili(wear_max*over/threshold)`, affects true
  - Band C (`over > threshold`): weapon_wear `wear_max`, affects_target false (too hard)
- `affects_target` gates ONLY mineable MATERIAL/armor wear -- NOT enemy HP. BOTH sides wear:
  the weapon always (by band), and the struck armor/material only when the hit affects it.

Split rule enforced: **HealthComponent = combat HP (ATK/DEF). DurabilityComponent = wear
(hardness).** A creature has a HealthComponent; a rock does not (you mine it, not ATK it).
`current_durability` is RUNTIME on the DurabilityComponent node -- never on a shared
resource (the sharing trap, `patterns/resource-driven-design.md`).

## Recipes / patterns that were wrong or awkward in practice

1. **A component export named `material` collides with the built-in `CanvasItem.material`
   and hard-fails to parse.** `@export var material: DurabilityComponent` on the Hurtbox
   (an Area2D) produced:
   `Parse Error: Member "material" redefined (original in native class 'Area2D')`
   -- a compile error that cascades to every dependent script. *Fix:* name component-
   reference exports so they cannot shadow a built-in Node property; here it became
   `material_durability`. `patterns/resource-driven-design.md` and `health-and-damage.md`
   should add a one-line caveat: an `@export` on a Node subclass shares the property
   namespace with the engine class (`material`, `modulate`, `position`, `scale`, ...), so
   prefix/suffix domain names for component refs.

2. **A tween-driven arc hitbox is timing-fragile to verify headless: the rotation TWEEN
   runs on IDLE-frame time while area overlap samples on PHYSICS frames.** Against a small
   target the whole 120-degree sweep can complete BETWEEN two physics samples -- the blade
   is logged at `rot 60` for frames 0-4 then `rot -60` at frame 5, never near 0, so the
   overlap window is never sampled and the hit is silently missed (isolated it lands; in a
   longer session it does not -- non-deterministic). This bites the `health-and-damage.md`
   "arc sweep on a pivot" note flagged in the B2 log. *Fix:* for a headless HIT assertion
   against a small Hurtbox, drive the LUNGE variant (blade held at a FIXED angle for the
   whole enabled window) instead of the sweeping arc -- a guaranteed overlap on every
   physics frame; reserve the arc for large targets (dummy r22, rocks 40x40) that tolerate
   coarse angular sampling. Also `await player.attack()` (let the engine run the swing at
   its natural cadence) and capture at the hit frame via the `hit_taken` signal, since
   `_knockback` decays over the awaited swing. The smoke test's leg a documents this inline.

3. **`--check-only --script res://...` needs `--path <proj>`.** `LLM_GUIDE.md`'s ladder
   step 2 shows `<godot_console> --headless --check-only --script res://path/to/script.gd`
   with no `--path`. Run bare, every `res://` script reports
   `Attempt to open script '...' resulted in error 'File not found'` (exit 1) because no
   project is mounted to resolve `res://`. *Fix:* the ladder command must be
   `<godot_console> --headless --path <proj> --check-only --script res://<script>.gd`.

## Bugs found and fixed WHILE writing the test (not source/recipe issues -- test-logic
bugs caught by running headless, recorded because they were non-obvious)

1. **Knockback drift between successive controlled hits on the same stationary
   target.** The new "creatures are never tool-gated" leg re-hits the SAME flesh enemy
   three times (sword, then axe, then pickaxe) from a fixed lunge position. Each hit
   applies the Hitbox's `knockback` (180 px/sec) to the enemy, and `stationary = true`
   only zeroes the enemy's OWN move velocity via friction -- it does not stop or reset
   the decaying `_knockback` impulse. Without re-zeroing `enemy.global_position` /
   `_move_velocity` / `_knockback` before EACH hit, the enemy drifted further from the
   fixed lunge-blade position on every hit; the second hit (axe) still landed by luck,
   the third (pickaxe) missed outright (0 damage recorded). Fixed by resetting all
   three before every controlled hit, not just the first.
2. **Two Hurtboxes at the same nominal position both firing on one strike.** The tree
   legs (m/n/o) position the Tree at `Vector2.ZERO` for testing, but the ObsidianRock
   was left at `Vector2.ZERO` from the prior leg (k) -- both Hurtboxes are 40x40 at the
   same spot, so a pickaxe strike meant to test "pickaxe whiffs on the tree" also
   overlapped the rock's Hurtbox, which is NOT gated for the pickaxe (MINE == MINE),
   producing a real Band C hit on the rock that read as "the pickaxe wore attacking the
   tree." Fixed by parking `obs_rock` at `(3000, 3000)` before the tree legs and
   bringing it back only for the leg that needs it (p). General lesson for this
   project's headless test style: EVERY object sharing a Hurtbox-sized footprint at the
   arena origin must be explicitly parked away before an unrelated leg runs, not just
   the ones the current leg's comment happens to mention.

## The Inventory API

```gdscript
class_name Inventory extends RefCounted

var slots: Array[ToolData] = [null, null, null, null, null, null]  # default size 6
var equipped_index: int = 0
var hotbar_unlocked: bool = false

func hotbar_size() -> int                    # mini(slots.size(), 10)
func equip_index(i: int) -> void              # direct jump, ignores lock, clamped
func cycle(delta: int) -> void                # +/-1, wraps hotbar window or full array
func equipped_tool() -> ToolData              # slots[equipped_index], or null
func add_tool(tool: ToolData) -> bool         # first-empty auto-populate
func sort() -> void                           # STUB -- see below
```

`cycle()`'s wrap uses `((equipped_index + delta) % n + n) % n` rather than a bare `%`
-- GDScript's `%` on a negative int follows truncated (C-style) division, so
`-1 % 10 == -1`, not `9`. Verified directly in the unit tests (the 10-key-ring legs).
`n` is `hotbar_size()` when locked, `slots.size()` when unlocked; number-key
`equip_index()` never consults `hotbar_unlocked` at all, so it always jumps straight
to the requested slot in either mode.

`sort()` is an intentional no-op this slice, per the user's own words in the design
doc ("if we hit 'sort' we will develop that later"). The doc's future behavior (tools
fill the hotbar first, non-tools push to the back/background) is documented inline as
a comment but not implemented -- no non-tool items exist yet to sort.

## Chunk constants + coord math (on WorldScale)

- `const CHUNK_TILES: int = 16`
- `const CHUNK_PX: float = TILE * CHUNK_TILES`  (= 640.0)
- `static func world_to_chunk(world_pos: Vector2) -> Vector2i`
    `return Vector2i(floori(world_pos.x / CHUNK_PX), floori(world_pos.y / CHUNK_PX))`
- `static func chunk_origin(coord: Vector2i) -> Vector2`
    `return Vector2(coord) * CHUNK_PX`

**Negative-coord handling (the bug-prone part).** GDScript integer `/` and `%`
TRUNCATE toward zero (`-1 % 16 == -1`, `-1 / 640 == 0`), which would wrongly map a
negative world position into chunk 0. `floori(world_pos.x / CHUNK_PX)` does a FLOAT
divide then floors toward -infinity, which is the correct chunk for all four quadrants.
Verified cases (all asserted): `(0,0)->(0,0)`, `(639,639)->(0,0)`, `(640,0)->(1,0)`,
`(-1,-1)->(-1,-1)`, `(-640,-640)->(-1,-1)`, `(-641,0)->(-2,0)`. Plus the round-trip
invariant `world_to_chunk(chunk_origin(c)) == c` for several coords including negatives.

(Chunk math lives on WorldScale, per the task's primary suggestion -- it stays a pure
static-function holder with no state, same shape as the existing TILE ruler.)

## ChunkData shape + to_dict/from_dict contract

`class_name ChunkData extends RefCounted` (NOT a Node -- the dormant form):
- `var coord: Vector2i`
- `var entries: Array[Dictionary]` -- each `{ "type": int(Kind), "local_pos": Vector2
  in [0,CHUNK_PX), "state": Dictionary }`. `state` is the small mutable payload
  (fresh mineral `{"integrity": 4}`, empty `{}` for a fresh tree/enemy). No Node refs,
  no Resource sub-objects -- plain value data only.
- `var dirty: bool = false` -- the future save trigger (C1 defines it; C3 flips it).
- `enum Kind { TREE, MINERAL, ENEMY }` -- extensible; never renumber (persists later).

`to_dict()/from_dict()` round-trip the whole chunk through a plain, store_var-/JSON-safe
Dictionary: `Vector2i`/`Vector2` are flattened to `[x, y]` arrays, and each entry's
`state` is DEEP-copied (`.duplicate(true)`) so the snapshot never aliases live entries.
Contract: **lossless** -- `from_dict(to_dict(x))` reproduces `x` field-for-field
(coord, dirty, entry count, and per-entry type/local_pos/state), and the restored state
dicts are independent copies (mutating one does not touch the other). All asserted.

## ChunkGenerator determinism guarantee

`static func generate(coord: Vector2i, world_seed: int) -> ChunkData`. DETERMINISTIC:
the local `RandomNumberGenerator`'s seed is derived reproducibly from `(coord,
world_seed)` via an integer XOR-mix with large odd constants (`_derive_seed`) -- NO
global RNG, NO `Time`/OS randomness. Same `(coord, world_seed)` always yields
byte-identical entries; a different coord OR seed shifts the derived seed and so yields
different content. Scatter per chunk (tunable local consts): 2-5 trees, 1-3 minerals,
a 35% chance of one enemy, at local positions in `[0, CHUNK_PX)`. This is the
regenerable baseline -- "persist only the DELTA from a regenerable baseline," so later
we save only chunks the player CHANGED. Asserted: two identical calls are equal;
different seed and different coord each diverge (proves it is seeded, not constant).

## Regen-on-revisit caveat (C3a, resolved by C3b)

There is no per-coord ChunkData store yet, so `_activate_chunk` always regenerates the
deterministic baseline. Revisiting a chunk therefore produces FRESH content -- a rock you
mined out reappears on return. This is acceptable for C3a and called out here for C3b, which
adds the store + delta write-back (`TODO(C3b)` in `_deactivate_chunk`) so a mutated chunk
persists its delta in-memory. Because activation stays deterministic, a revisit at least

## Recipe / pattern issues found

1. **Code-spawned scene instances must have their root `@export` overrides set BEFORE
   `add_child`.** `rock.gd`/`tree.gd` read their root exports in `_ready()` and push them onto
   child components; `_ready` fires on tree-entry, so setting `rock.integrity`/`rock.hardness`
   AFTER `add_child` would be too late (the child DurabilityComponent/Hurtbox would keep the
   scene defaults). This is the code-first equivalent of `main.tscn`'s instance-override path,
   which applies overrides before `_ready` for free. The spawner sets exports first, then the
   caller `add_child`s. Worth a one-line note wherever the "author-by-instance-override"
   pattern is documented: in code, "override then add", not "add then override".

2. **A from-state config assertion is only meaningful with a NON-default value.** The mineral's
   scene-default integrity (4), ChunkGenerator's fresh integrity (4), and DurabilityComponent's
   own default `max_durability` (4) all coincide, so a REAL generated rock reads 4 whether or
   not the entry->instance wiring works -- the assertion would pass on broken code. Fix used:
   drive a synthetic entry with `integrity: 2` through the exact same `ChunkContent.spawn()` and
   assert the instance reads 2, distinguishing wired-from-state vs scene-default. General
   testing lesson: to prove data flows A->B, pick a test value that differs from every default
   on the path.

3. **Distinguishing Tree from Rock in a type check needs Rock-before-StaticBody2D ordering.**
   Tree has no `class_name` (its name clashes with Godot's native `Tree` control), while `Rock`
   does and extends `StaticBody2D`. So the instance-counting loop must test `is Rock` (and `is
   Enemy`) BEFORE the catch-all `is StaticBody2D` = Tree, or every rock would be miscounted as a
   tree. No new bug -- just the consequence of tree.gd's documented native-class fallback.

## The store + write-back + capture design

- **`_store: Dictionary` (coord -> ChunkData), PERSISTENT.** Populated on first activation
  (generated baseline) and RETAINED across deactivation. This IS the in-memory dormant world:
  RefCounted DATA records, never Nodes. Grows with explored-chunk COUNT as cold data (the
  intended cost; the disk milestone caps it by swapping "generate" for "read disk else generate"
  at the SAME `_activate_chunk` branch).
- **`_content: Dictionary` (coord -> Array of nodes), ACTIVE-ONLY, index-aligned to entries.**
  A `null` slot marks a `gone` entry that was not spawned. Gives `_deactivate_chunk` the
  (entry -> live node) pairing to write state back.
- **`_activate_chunk`: load-from-store-else-generate, skip `gone`.** `var data = _store.get(coord)`;
  if null, generate + store. Spawn one node per entry via `ChunkContent.spawn(entry)`, skipping
  any entry whose `state.gone` is true (mined out / chopped / killed on a prior visit).
- **`_deactivate_chunk`: the write-back.** For each entry+node: if the node is no longer valid
  (`is_instance_valid` false -- destroyed and queue_freed during play) flag `state.gone = true`;
  else `ChunkContent.capture(node, entry)`. Set `data.dirty = true` on any change (the future
  save trigger). The ChunkData is NOT erased from `_store`; only the live Nodes are freed.
- **`ChunkContent.capture(node, entry) -> bool`** keeps the per-Kind "what state is durable"
  knowledge next to `spawn()` (Rule 1): MINERAL -> `entry.state.integrity =
  Material.current_durability` (or `gone` at 0); ENEMY -> `entry.state.hp =
  HealthComponent.current_health` (or `gone` at 0); TREE -> nothing (a felled tree is freed, so
  the manager's is_instance_valid path flags it gone). Returns true only when it actually
  changed the entry, so a pure traversal never dirties a chunk.
- **How gone / HP / integrity persist on RELOAD:** `spawn()` already applied `state.integrity`
  to a Rock's root export (C3a); C3b confirms it reads the possibly-reduced value. For ENEMY,
  `spawn()` now re-applies a stored `hp` on the enemy's `ready` signal (HealthComponent._ready
  resets current_health to max on tree-entry, so the override must run AFTER that -- the `ready`
  signal fires once, after all children _ready; connected CONNECT_ONE_SHOT so it self-cleans). A
  `gone` entry is never spawned (skipped in `_activate_chunk`).

## The respawn-policy design (INF = reload / finite = respawn-in-place)

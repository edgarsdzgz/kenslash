# Game Time + Day/Night Cycle -- Design (user-decided, 2026-07-19)

How in-game time works. The headline: game time is DETERMINISTIC because it is COUNTED from simulation
steps, never READ from the real world. Design track; not built yet.

## The decision
- **A single integer counter `game_tick`, advanced by ONE per FIXED simulation step** (Godot's
  `_physics_process` fixed timestep, the same cadence the player sim already runs on). NOT accumulated
  from `_process(delta)` render frames.
- **Time of day is a pure function of that counter:** `phase = f(game_tick)`. Same tick count -> same
  time of day, every run, every peer.
- **Independent of the real world by construction.** The game NEVER reads `Time`/`OS`/system clock for
  the clock. It only counts its own steps.

## Why counted-not-read = deterministic (not a rule violation)
Two different things both called "time":
- READING the real world (`Time.get_ticks_msec()`, `OS`, system clock) -- samples outside the game ->
  nondeterministic -> BANNED in the sim/generator (see the determinism rule in NOTES.md / CONVENTIONS).
- COUNTING our own fixed steps -- generated by the game itself -> byte-identical every replay -> FINE,
  and in fact the deterministic WAY to have time.
The day/night clock is the second kind, so it strengthens determinism rather than threatening it.

## The "IRL ratio" is a baked CONSTANT, not a clock read
"How many ticks approx= an IRL minute" is a DESIGN CONSTANT chosen for pacing, baked into config -- not
a runtime measurement. E.g. physics at 60 ticks/sec, and we DECIDE "1 in-game hour = 3600 ticks = 1 real
minute of play." The game never checks whether a real minute actually elapsed; it just knows "3600 ticks
since dawn -> 1am." The real world never enters the loop. (Exact ratio = TUNING, TBD.)

## The subtlety to get right (the one trap)
- FIXED step (`_physics_process`, 60 Hz) = framerate-independent, exact -> COUNT THIS.
- VARIABLE render delta (`_process(delta)`) drifts by framerate (60 vs 144 fps reach dawn at different
  points) and a frame hitch injects a big delta -> do NOT accumulate this into the authoritative clock.
  Fine for pure VFX only.
Rule of thumb: integer tick count, advanced in the fixed-step loop. Never the render delta.

## Payoffs (why this is a win, not a compromise)
- **One-integer save:** the whole world's time-of-day is just `game_tick`. Save it, reload, the cycle
  reconstructs. Fits "persist only the delta from a regenerable baseline."
- **Testable:** headless test does `advance N ticks; assert night` -- no waiting, no flake. If night
  changes spawns/behavior, that is assertable deterministically too.
- **Multiplayer-safe:** every peer agrees on the in-game hour because they ran the same tick count --
  mandatory if day/night affects spawns or AI.
- **Pause-correct:** tied to sim steps, so it stops on pause and never lurches after an alt-tab (unlike
  a wall-clock timer).

## Relationship to the memory-transmission save timer (design-lore.md)
- Day/night clock = GAME time: advances with play, pauses when paused -> tick-driven, INSIDE the
  deterministic sim.
- Memory-transmission autosave was framed as "every X IRL minutes." If that literally means real minutes
  while idle, it samples the wall clock -> keep it SYSTEM-layer with an injectable tick. BUT a cleaner
  option: define it as "every X minutes OF PLAY" -> then it too becomes tick-driven and deterministic,
  sharing the same honest game-time base as the day/night clock. (Recommendation: prefer game-time;
  decide when we build save.)

## Open questions / tuning (decide when we build)
1. Day length (ticks per full cycle) + the IRL-minute ratio -- pure tuning.
2. Number/definition of phases (dawn/day/dusk/night, or a continuous 0-1 time-of-day float derived from
   game_tick).
3. What day/night AFFECTS: lighting/ambience only, or gameplay (night = tougher/more enemies, worse
   visibility, different spawns)? Anything gameplay-affecting MUST stay on the deterministic tick.
4. Interaction with the REGION flag (design-environment.md #3): inside/caves ignore the surface
   day/night lighting; the flag is the hook to swap it.
5. Interaction with ELEVATION/isometric later (shadows, lighting angle) -- out of scope now.
6. Does the transmission-save cadence share game_tick (recommended) or stay wall-clock?

## Build order (when we get to it)
`game_tick` counter in the fixed-step loop -> time-of-day function -> save/restore the single integer ->
visual day/night (ambient tint, respects the region flag) -> (optional) gameplay hooks (night spawns/
visibility). Keep every gameplay-affecting hook on the deterministic tick.
